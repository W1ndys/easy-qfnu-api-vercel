# 完善系统日志记录 - 详细修改计划

## 一、现状审计

### 1.1 当前日志使用情况

| 文件                                   | 当前方式               | 问题                                                             |
| -------------------------------------- | ---------------------- | ---------------------------------------------------------------- |
| `main.go:34`                           | `log.Printf`           | 使用标准库 `log`，非结构化输出，与其他文件使用的 `slog` 不统一   |
| `middleware/logger.go`                 | `slog.Info/Warn/Error` | 使用 `slog` 默认 handler，无文件输出，无日志轮转，无 JSON 格式   |
| `middleware/recovery.go`               | `slog.Error`           | 同上，panic 日志仅输出到 stderr                                  |
| `middleware/auth.go`                   | 无日志                 | 鉴权失败时未记录任何日志（IP、路径等），安全审计盲区             |
| `middleware/cors.go`                   | 无日志                 | CORS 拒绝的请求无记录                                            |
| `services/zhjw/login.go:59,81`         | `slog.Info`            | 仅记录验证码重试和 OCR 结果，登录成功/失败无日志                 |
| `services/zhjw/grade.go:33`            | `slog.Info`            | 仅业务开始日志，缺少完成/失败/耗时日志                           |
| `services/zhjw/class_schedules.go:26`  | `slog.Info`            | 同上                                                             |
| `services/zhjw/exam_schedules.go:25`   | `slog.Info`            | 同上                                                             |
| `services/zhjw/selection_result.go:25` | `slog.Info`            | 同上                                                             |
| `services/zhjw/course_plan.go`         | 无日志                 | 完全没有日志记录                                                 |
| `services/zhjw/ocr.go`                 | 无日志                 | OCR 调用失败/成功无日志，仅通过 error 返回                       |
| `services/zhjw/client.go`              | 无日志                 | Cookie 过期检测拦截器无日志                                      |
| `internal_api/zhjw/*.go`               | 无日志                 | 所有 6 个 API Handler 均无日志（参数绑定失败、业务错误等无记录） |
| `router/router.go`                     | 无日志                 | 路由初始化、静态文件服务无日志                                   |

### 1.2 核心问题总结

1. **日志库混用**: `main.go` 使用 `log.Printf`，其余使用 `slog`，风格不统一
2. **无全局 Logger 初始化**: `slog` 使用默认 handler（纯文本输出到 stderr），无法配置输出格式/目标
3. **无日志文件输出**: 所有日志仅输出到 stderr/stdout，进程重启后日志丢失
4. **无日志轮转**: 即使写文件也无大小/时间轮转机制
5. **日志覆盖不完整**:
   - 鉴权失败无日志（安全风险）
   - API Handler 层完全无日志
   - 业务层仅有"开始"日志，无"完成/失败/耗时"日志
   - OCR 外部服务调用无日志
   - Cookie 过期拦截无日志
6. **无请求 ID**: 无法在分布式/并发场景下追踪单次请求的完整链路
7. **Gin 默认日志未关闭**: `gin.Default()` 自带 Logger 和 Recovery 中间件，与自定义的 `middleware.RequestLogger()` 和 `middleware.Recovery()` 重复

## 二、技术选型

### 推荐方案: `uber-go/zap`

**理由**:

- Go 生态最流行的高性能结构化日志库（GitHub 22k+ stars）
- 比 `slog` 快 4-10 倍（零分配设计）
- 原生支持 JSON / Console 两种输出格式
- 配合 `natefinch/lumberjack` 实现日志文件轮转
- 社区生态成熟，有 `gin-contrib/zap` 可直接替换 Gin 日志

### 依赖清单

| 包                                 | 用途         |
| ---------------------------------- | ------------ |
| `go.uber.org/zap`                  | 核心日志库   |
| `gopkg.in/natefinch/lumberjack.v2` | 日志文件轮转 |

> 注: 不引入 `gin-contrib/zap`，自行实现 Gin 中间件以保持与现有中间件风格一致，避免额外依赖。

### 新增环境变量

| 变量名            | 说明                                     | 默认值    |
| ----------------- | ---------------------------------------- | --------- |
| `LOG_LEVEL`       | 日志级别 (`debug`/`info`/`warn`/`error`) | `info`    |
| `LOG_FORMAT`      | 日志格式 (`json`/`console`)              | `console` |
| `LOG_FILE`        | 日志文件路径，为空则仅输出到 stdout      | 空        |
| `LOG_MAX_SIZE`    | 单个日志文件最大 MB                      | `100`     |
| `LOG_MAX_BACKUPS` | 保留的旧日志文件个数                     | `30`      |
| `LOG_MAX_AGE`     | 日志文件保留天数                         | `30`      |

## 三、Step-by-Step 修改计划

### Step 1: 新建日志初始化模块

**新建文件**: `pkg/logger/logger.go`

**功能**:

- 从环境变量读取配置
- 初始化 zap Logger（支持同时输出到 stdout 和文件）
- 配置 lumberjack 日志轮转
- 提供全局 `logger.L()` 和 `logger.S()`（Sugar Logger）访问函数
- 提供 `logger.Init()` 初始化函数
- 将 zap Logger 桥接到 slog 的 default handler，确保渐进式迁移时 slog 调用也走 zap

**关键逻辑**:

```
func Init() {
    1. 读取 LOG_LEVEL / LOG_FORMAT / LOG_FILE 等环境变量
    2. 创建 zapcore.Core:
       - encoder: 根据 LOG_FORMAT 选择 JSON 或 Console
       - writeSyncer: stdout + (可选) lumberjack 文件
       - level: 根据 LOG_LEVEL 设置
    3. 构建 zap.Logger, 设置 caller 信息
    4. 设置全局变量
    5. 替换 zap 全局 logger
}
```

---

### Step 2: 新建请求 ID 中间件

**新建文件**: `middleware/request_id.go`

**功能**:

- 为每个请求生成唯一 Request-ID（使用 UUID 或短 ID）
- 将 Request-ID 写入 `gin.Context` 和响应 Header `X-Request-ID`
- 后续中间件和 Handler 通过 `c.GetString("RequestID")` 获取

> 注: 使用 `crypto/rand` 生成 8 字节 hex 短 ID，避免引入 UUID 依赖。

---

### Step 3: 重写 `middleware/logger.go`

**修改文件**: `middleware/logger.go`

**变更**:

- 将 `slog` 替换为 `zap`
- 从 `gin.Context` 获取 Request-ID 写入日志字段
- 增加 `request_id`, `user_agent`, `body_size` 字段
- 保持原有的状态码分级逻辑（>=500 Error, >=400 Warn, 其余 Info）

---

### Step 4: 重写 `middleware/recovery.go`

**修改文件**: `middleware/recovery.go`

**变更**:

- 将 `slog.Error` 替换为 `zap.Error`
- 增加 `request_id` 字段
- 记录完整的 stack trace 到 zap 的 `stacktrace` 字段

### Step 5: 增强 `middleware/auth.go`

**修改文件**: `middleware/auth.go`

**变更**:

- 鉴权失败时记录 Warn 日志（包含 `request_id`, `ip`, `path`, `method`）
- 鉴权成功时记录 Debug 日志

### Step 6: 增强 `middleware/cors.go`

**修改文件**: `middleware/cors.go`

**变更**:

**变更**:

- 中间件注册顺序: `Recovery -> RequestID -> RequestLogger -> Cors`
- 记录路由初始化完成的 Info 日志

## 五、日志级别规范

| 级别      | 使用场景                                                                          |
| --------- | --------------------------------------------------------------------------------- |
| **Debug** | 请求参数详情、CORS 通过、鉴权通过、OCR 调用开始                                   |
| **Info**  | 服务启动、请求完成(2xx/3xx)、业务操作开始/完成、登录成功                          |
| **Warn**  | 客户端错误(4xx)、鉴权失败、Cookie过期、参数绑定失败、CORS拒绝、登录失败(用户原因) |
| **Error** | 服务器错误(5xx)、Panic恢复、外部服务调用失败(OCR)、HTML解析失败                   |

## 七、执行顺序建议

按 Step 1 -> 16 的顺序执行。Step 1（Logger 初始化）是所有后续步骤的基础，必须首先完成。Step 2（Request ID）是 Step 3-14 中日志字段的前置依赖。Step 8（Router 修改）应在 Step 3-6（中间件修改）之后执行，确保中间件已就绪。
